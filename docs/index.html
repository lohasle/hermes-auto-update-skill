Features
Everything you need.
Nothing you don't.

A single Hermes skill that handles deployment, service management, and ongoing upgrades — all automatically.

⚙

Zero-Config Auto-Detect

All paths — HERMES_HOME, venv, webui directory — are discovered automatically. No manual placeholders. No editing configs.

↻

Daily Auto-Upgrade

Two cron jobs run at 3:00 AM daily — one for hermes-agent, one for hermes-webui. Git pull, restart, done. Zero downtime.

■

Systemd, Not Docker

Native systemd service. Data lives in ~/.hermes, survives restarts and upgrades. No container rebuilds, no data loss.

⚡

Shared Agent Venv

WebUI runs on hermes-agent's venv — cron and tools modules are always available. No ModuleNotFoundError.

✓

Built-in Verification

Every step has a verification gate. /api/crons and /api/skills must return 200 before the skill reports success.

◎

Agent-Native

Designed as a Hermes skill. Any agent loads it via skill_manage and executes it autonomously. No human in the loop.

How it works
Three steps. One prompt.

Send the install prompt to any Hermes agent. It handles everything.

Step 1 — Deploy

Agent + WebUI

Git clone hermes-agent and hermes-webui. Create venv, pip install, systemd service. Verify APIs respond.

Step 2 — Schedule

Daily Cron Jobs

Register two cron jobs — one for agent updates, one for webui. Both run git fetch, compare commits, pull if needed.

Step 3 — Ship

Register & Done

Save as a Hermes skill for future reference. Data persists in ~/.hermes. Upgrades happen while you sleep.

Quick Start — Paste to Hermes Agent 
# Copy the prompt below and send to any Hermes agent

请从 GitHub 加载技能 hermes-auto-update：

1. 先执行: git clone https://github.com/lohasle/hermes-auto-update-skill.git /tmp/hermes-auto-update-skill
2. 读取 /tmp/hermes-auto-update-skill/SKILL.md，按照其中的步骤逐一执行
3. 执行完毕后，用 skill_manage 将 SKILL.md 注册为技能 hermes-auto-update
4. 创建两个每日自动升级 cron job（详见 SKILL.md Step 6）

关键提醒：
- 所有路径自动探测，不要问用户
- 不用 Docker
- 必须用 hermes-agent 的 venv python 跑 webui
- 验证 /api/crons 和 /api/skills 都正常才算成功
Why not Docker
Native beats containerized.

Docker caused data loss, module errors, and slow rebuilds. Systemd is faster, simpler, and your data survives.

Docker Native systemd
Data persistence Container rebuild loses data ~/.hermes survives forever
cron / tools API 500 errors without mount hacks Agent venv, zero errors
Upgrade speed Rebuild image, minutes git pull + restart, seconds
Configuration Mount paths, UID/GID mapping Auto-detect, zero config
Resource usage Docker daemon overhead Minimal, bare metal
